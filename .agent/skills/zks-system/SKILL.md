# zks-system

#### User-Space Friendly (Права доступа)

- **Цель:** Не использовать повышение привилегий (`sudo`/`doas`) там, где это не нужно.
- **Squashfuse:** Для монтирования **нешифрованных** squashfs-образов в порте на Rust используется FUSE-утилита `squashfuse`, которая не требует root-прав. Подход, который ранее использовался в fish-версии (`sudo mount`/`sudo umount`) считается устаревшим **для plain-режима**.
- **Исключение (LUKS):** Для зашифрованных LUKS-контейнеров `squashfuse` **неприменим** технически, т.к. `cryptsetup` создает блочное устройство `/dev/mapper/*`, принадлежащее `root`. В этом случае допускается использование `sudo mount` / `sudo umount`.
- **Суть:** Если вы восстанавливаете свои личные файлы в домашнюю папку, утилита не должна требовать root-прав. Root должен запрашиваться только для системных путей или для работы с LUKS.
- **Точки монтирования:** По умолчанию авто-генерируемые точки монтирования создаются локально в `/tmp/0k-cache-<uid>/` для обеспечения стабильности при работе с сетевыми дисками и избегания ограничений FUSE-на-FUSE.

#### 1. Архитектура "User Namespace Bind Strategy"
(Актуально только для режима работы от текущего пользователя, когда все файлы доступны для чтения)

* **Реализация:** `0k` создает изолированное окружение (User Namespace) для безопасного монтирования (`mount --bind`).
* **Ограничение:** Внутри namespace файлы пользователя (UID 1000) видятся как `nobody` или принадлежащие внутреннему `root`, если маппинг не идеален.

#### 2. Стратегия Владения (Ownership Strategy)

* **Freeze / Create:**
    * **Вариант А (Доступ есть):** Если у пользователя есть права на чтение всех файлов, упаковка (`mksquashfs`) может запускаться от его имени (внутри или снаружи UserNS). В архиве файлы будут принадлежать текущему пользователю.
    * **Вариант Б (Системные/Чужие файлы):** Если требуются root-права (системные пути или файлы другого пользователя), `0k` запрашивает эскалацию (`sudo`). Упаковка (`mksquashfs`) запускается от настоящего `root` (без UserNS, так как root и так всё может). В архиве сохраняются **оригинальные** UID/GID владельцев.
    
* **Mount:**
    * В режиме без флага `-e` всегда используется `squashfuse` (без root).
    * При работе с LUKS в режиме с `-e` squashfuse использовать не нужно, используется системная утилита `mount`

* Mount:   
    * Файлы видны с их записанными UID/GID.
    * Если файл принадлежит `root` и имеет права `600`, пользователь увидит его в каталоге, но получит `Permission denied` при попытке чтения. Это штатное поведение Linux.




* **Unfreeze:**
    * **По умолчанию:** Пытается распаковать (`rsync`) от имени пользователя.
    * **Эскалация:** Если в архиве обнаружены файлы с чужим UID/GID или недоступные для чтения права, операция переключается в режим **Sudo Mode** (`sudo rsync`), предварительно уведомив пользователя и запросив пароль. Это гарантирует корректное восстановление владельцев.
    * **Обратная совместимость (Legacy):** Если в `list.yaml` отсутствует поле `privilege_mode` (старый формат):
        * Утилита сообщает пользователю: "Archive is in legacy format. Recommendation: Repackage after restoring."
        * Явно спрашивает у пользователя режим восстановления: `[User/Root]`.

#### 3. Архитектурные ограничения (Technical Constraints)

1. **Pipeline Шифрования (Parent-Child):**
   * При использовании User Namespace (`unshare`), команда `sudo` внутри него не работает.
   * **Решение:** Процесс **сборки** (UserNS) пишет результат в `stdout`. Процесс **родитель** (HostNS) читает этот `stdout` и передает его в процесс **шифрования** (`sudo cryptsetup`), который работает в Host Namespace.
   
2. **Очистка (Stale Mounts):**
   * При аварийном завершении `0k` точки монтирования в `/tmp/0k-cache-<uid>/` могут зависнуть.
   * **Решение:** При старте приложение проверяет наличие "мертвых" маунтов в своей кэш-директории (`/tmp/0k-cache-<uid>/`) и очищает их:
     - **Plain (FUSE):** `fusermount -u` — без sudo.
     - **LUKS (системные маунты):** `sudo umount` + `sudo cryptsetup close` — с запросом пароля sudo.
   * **Ограничения:**
     - Перед размонтированием проверяется, что PID процесса-владельца (если он запомнен в lock-файле) больше не существует.
     - Если User Namespace "умер", его bind-маунты автоматически очищаются ядром.

3. **Squashfuse Context:**
   * Запуск FUSE внутри User Namespace требует сложной настройки (`/dev/fuse`, mounting fuse fs).
   * **Решение:** Команды `check` и монтирование для восстановления лучше выполнять в Host Namespace от имени пользователя, так как `squashfuse` не требует root.

#### 4. Смена "Сборочного цеха" (Staging Area)

Мы отказываемся от использования глобальной папки `/tmp` для сборки структуры архива в пользу пользовательского кэша.

* **Fish (Legacy):** Использовала `/tmp/zks_build_<uuid>`. Это небезопасно (доступно другим пользователям) и захламляет системный tmp.
* **Rust (Target):**
    * Используй директорию `/tmp/0k-cache-<uid>/build_<unix-time>_<random>/` (где `<uid>` — ID текущего пользователя).
    * Эта папка принадлежит пользователю, доступ к ней ограничен правами `0700`, она не захламляет домашний каталог и не требует сетевых ресурсов.

##### Механизм защиты от удаления кэша во время сборки

> [!IMPORTANT]
> Обычный пользователь **не может** выполнять `mount -t tmpfs` в Host Namespace (требуется root). Поэтому защита реализуется через file locking (`flock`), а не через монтирование.

1. **Подготовка (Host Namespace):**
    
    - `0k` создает базовую директорию сессии: `/tmp/0k-cache-<uid>/<session_id>/` (где `<session_id>` — формат `build_<unix-time>_<random>`).
        
    - **Защитный слой (flock):** Создается файл `/tmp/0k-cache-<uid>/<session_id>/.lock` и на него устанавливается эксклюзивная блокировка через `flock(LOCK_EX)`.
        
    - Перед любым рекурсивным удалением каталогов в кэше утилита должна проверять наличие `.lock` файла и пытаться получить блокировку. Если блокировка занята — директория используется, удаление прерывается с ошибкой.
        
    - Внутри директории сессии воссоздается дерево каталогов и создаются пустые файлы-заглушки (`touch`).
        
2. **Изоляция (Enter Namespace):**
    
    - `0k` делает `unshare(CLONE_NEWUSER | CLONE_NEWNS)`.
        
    - Процесс получает собственную копию таблицы монтирования.
        
    - **Важно:** Bind mounts создаются **после** входа в namespace, чтобы не "утечь" в Host NS.
        
3. **Связывание (Bind Mounts — внутри Namespace):**
    
    - Внутри namespace устанавливается флаг `MS_PRIVATE` / `MS_REC`, чтобы bind mounts не распространялись на Host NS.
        
    - Процесс выполняет `mount --bind /real/path $STUBS_DIR/file_or_dir_stub`.
        
    - `mksquashfs` также запускается **внутри** этого namespace.
        
4. **Упаковка:**
    
    - Запускается `mksquashfs` на директорию `$STUBS_DIR`.
        
5. **Очистка и Разблокировка:**
    
    - После завершения упаковки процесс выходит из namespace. Bind mounts автоматически уничтожаются ядром при завершении namespace.
        
    - **Разблокировка:** При закрытии файлового дескриптора `.lock` блокировка `flock` снимается автоматически.
        
    - _Только после этого папка становится "разблокированной" и её можно безопасно удалить из кэша._


#### 5. Алгоритм сборки (Stub & Bind)

**НИ ПРИ КАКИХ ОБСТОЯТЕЛЬСТВАХ** не используем сборку каталога через копирование (`cp`) или хардлинки (`ln`). Мы используем стратегию **"Заглушка и Монтирование"**. Это обеспечивает **Zero-Copy** (нулевое копирование данных) и мгновенную скорость работы даже между разными физическими дисками.

**Пошаговый алгоритм реализации Freeze:**

1.  **Подготовка (Host Namespace):**
    * `0k` парсит аргументы и строит список целей.
    * В папке `/tmp/0k-cache-<uid>/.../to_restore/` воссоздается дерево каталогов.
    * Для каждого целевого файла создается **пустой файл-заглушка** (`touch`).
    * Для каждой целевой директории создается **пустая папка**.
    * *На этом этапе данные не читаются, место на диске не тратится.*

2.  **Изоляция (Enter Namespace):**
    * `0k` делает `unshare(CLONE_NEWUSER | CLONE_NEWNS)`.
    * Теперь процесс "думает", что он root.

3.  **Связывание (Bind Mounts):**
    * Процесс пробегается по списку заглушек и выполняет `mount --bind /real/path /cache/stub`.
    * Теперь в папке кэша вместо пустых заглушек "магически" появляется реальное содержимое файлов.

4.  **Упаковка:**
    * Запускается `mksquashfs` на папку кэша. Утилита видит реальные данные через точки монтирования.

5.  **Очистка:**
    * При выходе из процесса namespace уничтожается ядром. Все точки монтирования исчезают автоматически.
    * Остается только удалить папку с пустыми заглушками в кэше. Это быстро и безопасно, но на всякий случай мы используем логику, аналогичную `rm-if-empty` (удаляем только если внутри нет реальных данных, чтобы случайно не задеть файлы пользователя, если namespace не отмонтировался).

#### 6. Обработка привилегий (Sudo)
При повышение привилегий используется точечно и только для тех операций, где это реально необходимо.
**Приоритет эскалаторов:** `sudo`, `doas`, `pkexec`, `run0`.
При каждом запросе на повышение привилегий утилита чётко объясняет пользователю (на английском языке), для какой именно цели они запрашиваются.

> [!IMPORTANT]
> **Принцип "Error Whitelist" для Эскалации:**
> Автоматический запрос на повышение привилегий (sudo prompt) разрешено показывать **ТОЛЬКО** если произошла подтвержденная ошибка прав доступа (`PermissionDenied`, `EACCES`, `EPERM`).
> **ЗАПРЕЩЕНО** запрашивать sudo "на всякий случай" при любых других ошибках (например, "File exists", "Disk full", "I/O Error").
> Если ошибка не связана с правами — утилита должна завершиться (паника или graceful exit) и вывести текст ошибки пользователю. Это критично для корректной работы тестов и CI.

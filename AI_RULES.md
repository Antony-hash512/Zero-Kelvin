## 2. Описание оригинальных fish-функций
Это CLI-утилита для **выгрузки данных** (Cold Storage) в среде Arch Linux.
В отличие от традиционных систем резервного копирования (цель которых — избыточность), `zks` предназначен для «заморозки» пользовательских данных: переноса их с локального хоста на внешнее хранилище (NAS/HDD) для освобождения дискового пространства, сохраняя при этом возможность легко «разморозить» (восстановить) их обратно.

**Цель проекта:** Создать надежную, быструю и безопасную CLI-утилиту для создания оффлоадов (полных сжатых бэкапов), использующую технологии SquashFS, ZSTD для сжатия, LUKS-шифрование (опционально) и Rclone для создания sqfs-offload'ов сразу на сетевых хранилищах без монтирования сетевых каталогов т.к. передачах данных по сети через смонтированный сетевой каталог происходит медленнее  (это тоже опциональная возможность и взаимоисключающая с LUKS-шифрованием, т.к. LUKS требует блочного доступа (Random Access), а `rclone rcat` работает с потоком данных. Другие методы поточного шифрования (GPG/Age) технически возможны, но выходят за рамки текущего ТЗ).

**Философия и цели:**
* **Offloading (Выгрузка):** Концептуально это перемещение данных, а не просто копирование.
    * *Техническая реализация:* Данные копируются в образ SquashFS, проверяются, и только после успешного создания архива исходные файлы удаляются с локального диска (опционально, по команде пользователя или флагом, по-умолчанию не удаляются, остаются для проверки и удаления вручную ).
* ** Опционально: Zero-Knowledge (Приватность):** Архивы должны быть «черными ящиками». Без ключа дешифрования ни метаданные, ни имена файлов, ни структура каталогов не должны быть видны.
    * *Реализация:* Сама `zks` не реализует криптографию. Она делегирует эту задачу зависимости `squash_manager`, передавая ей флаг `-e` (или `--encrypt`), только если такой же флаг был передан самой `zks`.
* **Contextual Sets (Контекстные наборы):** Возможность сгруппировать разрозненные пути (например, `/home/user/project` + `/etc/project.conf`) под одним идентификатором архива (ID).
* **Low Friction (Минимум трения):** Процесс восстановления должен быть автоматизированным. Утилита сама знает исходные пути файлов благодаря внутреннему манифесту.
* **Мгновенный доступ:** Использование SquashFS позволяет примонтировать архив и просмотреть содержимое без полной распаковки.

### Философия
#### Offloading (Выгрузка), а не Backup (Резервирование)

Это  ключевое отличие от стандартных решений (TimeShift, Borg, Restic).

- **Цель:** Освобождение дискового пространства на ноутбуке (Arch Linux).
- **Суть:** Не столько резервные копии (хотя на NAS с RAID5-HDD хранить всё-равно безопаснее, чем на SSD ноута), сколько «замораживание» проектов или данных, которые **не нужны прямо сейчас**, но могут понадобиться в будущем.
- **Действие:** Данные переносятся на внешний носитель (NAS или HDD) с локального диска.
#### Минимизация «телодвижений» (Low Friction Restore)

- **Цель:** Восстановление данных должно быть почти таким же простым, как `Ctrl+Z` (образно говоря).
- **Суть:** Утилита должна сама помнить, **откуда** были взяты файлы.
- **Решение:**
    - Пользователь не должен вручную указывать: «Распакуй этот архив в `/home/user/project`».
    - Утилита читает внутренний манифест (`list.yaml`) и сама предлагает: «Этот архив содержит файлы из `/etc/nginx` и `/home/user`. Вернуть их на место?».
    - Использование уникальных ID (имя хоста + дата + тег) позволяет быстро находить нужный "слепок" системы.
#### Гибкость и Атомарность (Contextual Sets)

- **Цель:** Умение работать с логически связанными, но физически разрозненными данными.
- **Суть:** Проект — это часто не одна папка. Это «код в `/home` + конфиг в `/etc` + база в `/var`»
- **Требование:** Утилита должна уметь собрать эти разбросанные пути в **один** архивный файл. И при восстановлении разложить их обратно по полочкам, не создавая конфликтов имен (решено через структуру нумерованных контейнеров `1/`, `2/`).

#### User-Space Friendly (Права доступа)

- **Цель:** Не использовать повышение привилегий (`sudo`/`doas`) там, где это не нужно.
- **Squashfuse:** Для монтирования **нешифрованных** squashfs-образов в порте на Rust используется FUSE-утилита `squashfuse`, которая не требует root-прав. Подход, который ранее использовался в fish-версии (`sudo mount`/`sudo umount`) считается устаревшим **для plain-режима**.
- **Исключение (LUKS):** Для зашифрованных LUKS-контейнеров `squashfuse` **неприменим** технически, т.к. `cryptsetup` создает блочное устройство `/dev/mapper/*`, принадлежащее `root`. В этом случае допускается использование `sudo mount` / `sudo umount`.
- **Суть:** Если вы восстанавливаете свои личные файлы в домашнюю папку, утилита не должна требовать root-прав. Root должен запрашиваться только для системных путей или для работы с LUKS.
- **Точки монтирования:** По умолчанию авто-генерируемые точки монтирования создаются локально в `/tmp/0k-cache-<uid>/` для обеспечения стабильности при работе с сетевыми дисками и избегания ограничений FUSE-на-FUSE.
#### KISS и Native (Arch Way)

- **Цель:** Простота, прозрачность и интеграция.
- **Суть:**
    - Никаких демонов, висящих в фоне.
    - Никаких сложных баз данных (всё хранится в файлах и внутри самих архивов).
    - Написано на **Fish**, для быстрого доступа к функционалу из данной современной оболочки.
    - Использование стандартных Linux-инструментов (`rsync`, `mksquashfs`, `cryptsetup`), чтобы данные можно было восстановить «руками», даже если утилита `zks` исчезнет.
        
**Итог одной фразой:**

Нужно создать **персональную, шифрованную «криокамеру»** для проектов и данных, которая позволяет одним движением освобождать место на ноутбуке и так же легко возвращать всё обратно, сохраняя структуру путей и права доступа.

## 3. Порт на Rust

### Отличия порта от реализации на fish

Хотя порт должен сохранять обратную совместимость по флагам CLI и структуре архивов, внутренняя архитектура работы с файловой системой и привилегиями кардинально меняется.

Главная цель изменений: **Работа без Root (Rootless Freeze)** всегда, когда это возможно. Права суперпользователя могут потребоваться для восстановления системных файлов или для архивации файлов, недоступных для чтения, но не для работы с обычными пользовательскими файлами.

#### 1. Архитектура "User Namespace Bind Strategy"
(Актуально только для режима работы от текущего пользователя, когда все файлы доступны для чтения)

* **Реализация:** `0k` создает изолированное окружение (User Namespace) для безопасного монтирования (`mount --bind`).
* **Ограничение:** Внутри namespace файлы пользователя (UID 1000) видятся как `nobody` или принадлежащие внутреннему `root`, если маппинг не идеален.

#### 2. Стратегия Владения (Ownership Strategy)

* **Freeze / Create:**
    * **Вариант А (Доступ есть):** Если у пользователя есть права на чтение всех файлов, упаковка (`mksquashfs`) может запускаться от его имени (внутри или снаружи UserNS). В архиве файлы будут принадлежать текущему пользователю.
    * **Вариант Б (Системные/Чужие файлы):** Если требуются root-права (системные пути или файлы другого пользователя), `0k` запрашивает эскалацию (`sudo`). Упаковка (`mksquashfs`) запускается от настоящего `root` (без UserNS, так как root и так всё может). В архиве сохраняются **оригинальные** UID/GID владельцев.
    
* **Mount:**
    * В режиме без флага `-e` всегда используется `squashfuse` (без root).
    * При работе с LUKS в режиме с `-e` squashfuse использовать не нужно, используется системная утилита `mount`

* Mount:   
    * Файлы видны с их записанными UID/GID.
    * Если файл принадлежит `root` и имеет права `600`, пользователь увидит его в каталоге, но получит `Permission denied` при попытке чтения. Это штатное поведение Linux.




* **Unfreeze:**
    * **По умолчанию:** Пытается распаковать (`rsync`) от имени пользователя.
    * **Эскалация:** Если в архиве обнаружены файлы с чужим UID/GID или недоступные для чтения права, операция переключается в режим **Sudo Mode** (`sudo rsync`), предварительно уведомив пользователя и запросив пароль. Это гарантирует корректное восстановление владельцев.
    * **Обратная совместимость (Legacy):** Если в `list.yaml` отсутствует поле `privilege_mode` (старый формат):
        * Утилита сообщает пользователю: "Archive is in legacy format. Recommendation: Repackage after restoring."
        * Явно спрашивает у пользователя режим восстановления: `[User/Root]`.

#### 3. Архитектурные ограничения (Technical Constraints)

1. **Pipeline Шифрования (Parent-Child):**
   * При использовании User Namespace (`unshare`), команда `sudo` внутри него не работает.
   * **Решение:** Процесс **сборки** (UserNS) пишет результат в `stdout`. Процесс **родитель** (HostNS) читает этот `stdout` и передает его в процесс **шифрования** (`sudo cryptsetup`), который работает в Host Namespace.
   
2. **Очистка (Stale Mounts):**
   * При аварийном завершении `0k` точки монтирования в `/tmp/0k-cache-<uid>/` могут зависнуть.
   * **Решение:** При старте приложение проверяет наличие "мертвых" маунтов в своей кэш-директории (`/tmp/0k-cache-<uid>/`) и очищает их:
     - **Plain (FUSE):** `fusermount -u` — без sudo.
     - **LUKS (системные маунты):** `sudo umount` + `sudo cryptsetup close` — с запросом пароля sudo.
   * **Ограничения:**
     - Перед размонтированием проверяется, что PID процесса-владельца (если он запомнен в lock-файле) больше не существует.
     - Если User Namespace "умер", его bind-маунты автоматически очищаются ядром.

3. **Squashfuse Context:**
   * Запуск FUSE внутри User Namespace требует сложной настройки (`/dev/fuse`, mounting fuse fs).
   * **Решение:** Команды `check` и монтирование для восстановления лучше выполнять в Host Namespace от имени пользователя, так как `squashfuse` не требует root.

#### 4. Смена "Сборочного цеха" (Staging Area)

Мы отказываемся от использования глобальной папки `/tmp` для сборки структуры архива в пользу пользовательского кэша.

* **Fish (Legacy):** Использовала `/tmp/zks_build_<uuid>`. Это небезопасно (доступно другим пользователям) и захламляет системный tmp.
* **Rust (Target):**
    * Используй директорию `/tmp/0k-cache-<uid>/build_<unix-time>_<random>/` (где `<uid>` — ID текущего пользователя).
    * Эта папка принадлежит пользователю, доступ к ней ограничен правами `0700`, она не захламляет домашний каталог и не требует сетевых ресурсов.

##### Механизм защиты от удаления кэша во время сборки

> [!IMPORTANT]
> Обычный пользователь **не может** выполнять `mount -t tmpfs` в Host Namespace (требуется root). Поэтому защита реализуется через file locking (`flock`), а не через монтирование.

1. **Подготовка (Host Namespace):**
    
    - `0k` создает базовую директорию сессии: `/tmp/0k-cache-<uid>/<session_id>/` (где `<session_id>` — формат `build_<unix-time>_<random>`).
        
    - **Защитный слой (flock):** Создается файл `/tmp/0k-cache-<uid>/<session_id>/.lock` и на него устанавливается эксклюзивная блокировка через `flock(LOCK_EX)`.
        
    - Перед любым рекурсивным удалением каталогов в кэше утилита должна проверять наличие `.lock` файла и пытаться получить блокировку. Если блокировка занята — директория используется, удаление прерывается с ошибкой.
        
    - Внутри директории сессии воссоздается дерево каталогов и создаются пустые файлы-заглушки (`touch`).
        
2. **Изоляция (Enter Namespace):**
    
    - `0k` делает `unshare(CLONE_NEWUSER | CLONE_NEWNS)`.
        
    - Процесс получает собственную копию таблицы монтирования.
        
    - **Важно:** Bind mounts создаются **после** входа в namespace, чтобы не "утечь" в Host NS.
        
3. **Связывание (Bind Mounts — внутри Namespace):**
    
    - Внутри namespace устанавливается флаг `MS_PRIVATE` / `MS_REC`, чтобы bind mounts не распространялись на Host NS.
        
    - Процесс выполняет `mount --bind /real/path $STUBS_DIR/file_or_dir_stub`.
        
    - `mksquashfs` также запускается **внутри** этого namespace.
        
4. **Упаковка:**
    
    - Запускается `mksquashfs` на директорию `$STUBS_DIR`.
        
5. **Очистка и Разблокировка:**
    
    - После завершения упаковки процесс выходит из namespace. Bind mounts автоматически уничтожаются ядром при завершении namespace.
        
    - **Разблокировка:** При закрытии файлового дескриптора `.lock` блокировка `flock` снимается автоматически.
        
    - _Только после этого папка становится "разблокированной" и её можно безопасно удалить из кэша._


#### 5. Алгоритм сборки (Stub & Bind)

**НИ ПРИ КАКИХ ОБСТОЯТЕЛЬСТВАХ** не используем сборку каталога через копирование (`cp`) или хардлинки (`ln`). Мы используем стратегию **"Заглушка и Монтирование"**. Это обеспечивает **Zero-Copy** (нулевое копирование данных) и мгновенную скорость работы даже между разными физическими дисками.

**Пошаговый алгоритм реализации Freeze:**

1.  **Подготовка (Host Namespace):**
    * `0k` парсит аргументы и строит список целей.
    * В папке `/tmp/0k-cache-<uid>/.../to_restore/` воссоздается дерево каталогов.
    * Для каждого целевого файла создается **пустой файл-заглушка** (`touch`).
    * Для каждой целевой директории создается **пустая папка**.
    * *На этом этапе данные не читаются, место на диске не тратится.*

2.  **Изоляция (Enter Namespace):**
    * `0k` делает `unshare(CLONE_NEWUSER | CLONE_NEWNS)`.
    * Теперь процесс "думает", что он root.

3.  **Связывание (Bind Mounts):**
    * Процесс пробегается по списку заглушек и выполняет `mount --bind /real/path /cache/stub`.
    * Теперь в папке кэша вместо пустых заглушек "магически" появляется реальное содержимое файлов.

4.  **Упаковка:**
    * Запускается `mksquashfs` на папку кэша. Утилита видит реальные данные через точки монтирования.

5.  **Очистка:**
    * При выходе из процесса namespace уничтожается ядром. Все точки монтирования исчезают автоматически.
    * Остается только удалить папку с пустыми заглушками в кэше. Это быстро и безопасно, но на всякий случай мы используем логику, аналогичную `rm-if-empty` (удаляем только если внутри нет реальных данных, чтобы случайно не задеть файлы пользователя, если namespace не отмонтировался).

#### 6. Обработка привилегий (Sudo)
При повышение привилегий используется точечно и только для тех операций, где это реально необходимо.
**Приоритет эскалаторов:** `sudo`, `doas`, `pkexec`, `run0`.
При каждом запросе на повышение привилегий утилита чётко объясняет пользователю (на английском языке), для какой именно цели они запрашиваются.

> [!IMPORTANT]
> **Принцип "Error Whitelist" для Эскалации:**
> Автоматический запрос на повышение привилегий (sudo prompt) разрешено показывать **ТОЛЬКО** если произошла подтвержденная ошибка прав доступа (`PermissionDenied`, `EACCES`, `EPERM`).
> **ЗАПРЕЩЕНО** запрашивать sudo "на всякий случай" при любых других ошибках (например, "File exists", "Disk full", "I/O Error").
> Если ошибка не связана с правами — утилита должна завершиться (паника или graceful exit) и вывести текст ошибки пользователю. Это критично для корректной работы тестов и CI.

### Архитектурные Принципы

#### Внешние Утилиты (Runtime Dependencies)

Мы **не** переписываем SquashFS или LUKS на Rust. Мы управляем ими.
Утилита должна полагаться на наличие в системе:

* `mksquashfs` / `unsquashfs` / `squashfuse`
* `cryptsetup` (требует root через sudo или doas)
* `tar2sqfs` 
* `rclone`
* `fusermount` (для размонтирования FUSE)

> [!IMPORTANT]
> **Проверка зависимостей при старте:**
> При запуске любой команды утилита должна проверять наличие **необходимых** для этой команды внешних утилит:
> - `0k-core create` → требует `mksquashfs`, опционально `cryptsetup` (если `-e`)
> - `0k-core mount` → требует `squashfuse` (plain) или `cryptsetup` + `mount` (LUKS)
> - `0k-core umount` → требует `fusermount` (plain) или `umount` + `cryptsetup` (LUKS)
>
> Если утилита не найдена — программа должна завершаться с понятной ошибкой:
> ```
> Error: Required tool 'squashfuse' not found in PATH.
> Install it via: sudo pacman -S squashfuse
> ```

#### Два режима работы (Взаимоисключающие)

1. **Secure Mode (LUKS):**
* Создание архива -> Шифрование через `cryptsetup` (в файл-контейнер) -> Запись на диск.
* Требует свободного места на локальном диске или монтирования (FUSE).


2. **Streaming Mode (`--direct-rclone`):**
* Пайплайн: `mksquashfs (stdout) | rclone rcat (stdin)`.
* **Нулевая запись на диск ноутбука.**
* Несовместим с LUKS (на данном этапе).

При этом возможна конечно и работа и без одного из этих режимов: просто создание сжатого незашифрованно SquashFS-образа в указанном каталоге (смонтированном или нет). Назовём этот режим и без LUKS или без Rclone как "plain", соответсветно режим без Rclone запись файла в каталог локальной ФС, а не на удалённую машину по сети это "plain/LUKS". Пока что в fish-функциях реализован имеено он. 

Streaming Mode через Rclone в fish-функциях пока не реализован. Его реализация запланирована в будущих версиях Rust-порта и будет осуществляться только после полного переноса текущего функциона из Fish в Rust. При этом писать код порта Rust нужно с учётом того, что такая возмонжность будет добавлена в дальнейшем.


#### Тестирование и Mocking (Критически важно)

* **Unit-тесты:** Вся логика, которая *формирует* команды, должна быть протестирована без реального запуска.
* Используй паттерн `CommandExecutor` (Trait) и макрос `#[automock]` из `mockall`, чтобы подменять реальный вызов `std::process::Command` на Mock-объект в тестах. Мы должны проверять, *правильные ли флаги* мы передаем в `cryptsetup`, не запуская его.

### Методология разработки (Strict TDD)

Разработка ведется строго по методологии **Test Driven Development (Red-Green-Refactor)**.
Ты не имеешь права писать реализацию функционала, пока под него не написан unit-тест.

**Алгоритм работы над любой задачей:**

1.  **Создай тест (Red):**
    * Напиши unit-тест, который описывает желаемое поведение (например, "функция должна вернуть корректную строку команды `mksquashfs`").
    * Убедись, что тест не проходит (или код не компилируется, так как функции еще нет).
2.  **Реализуй минимум (Green):**
    * Напиши минимально необходимый код, чтобы тест прошел. Не пиши лишней логики "про запас".
3.  **Улучши (Refactor):**
    * Приведи код в соответствие со стандартами стиля, убери дублирование, но следи, чтобы тесты оставались зелеными.

**Специфика TDD и уровни тестирования:**

Поскольку мы работаем с внешними утилитами, важно четко разделять типы тестов:

1.  **Unit-тесты (в папке `src/`):**
    * **Цель:** Проверка внутренней логики (парсинг аргументов, формирование команд, обработка YAML).
    * **Запрет:** Здесь **ЗАПРЕЩЕНО** запускать реальные внешние процессы (`mksquashfs`, `cryptsetup`) или выполнять долгие операции ввода-вывода.
    * **Инструмент:** Используй крейт `mockall` и макрос `#[automock]` для автоматической генерации Mock-объектов трейта `CommandExecutor`, чтобы проверить, что Rust-код *намеревается* запустить правильную команду с правильными флагами. Мы тестируем намерения, а не побочные эффекты. Не пиши моки вручную.
    * **Важно для mockall:** Если метод трейта принимает аргументы по ссылке с временем жизни (например, `&[&str]` или `&Path`), используй `.withf(...)` вместо `.with(eq(...))`. Предикаты `eq` плохо работают с Generic Lifetimes, вызывая ошибки компиляции `implementation of Borrow is not general enough`. Используй обычное сравнение `==` внутри замыкания `.withf`.
2.  **Интеграционные тесты (в папке `tests/`):**
    * **Цель:** Проверка реальной работы ("Smoke Tests" на маленьких файлах).
    * **Разрешение:** Здесь **НУЖНО** работать с реальными файлами (через `tempfile`) и запускать реальные бинарники.
    * **Приоритет:** Такие тесты пишутся после того, как Unit-тесты прошли (Green).


Для запуска интеграционных тестов используй `fish tests/run_shell_tests.fish`, а НЕ ЗАПУСКАЙ bats-файлы по отдельности.

**Правило:** Если в твоем ответе есть код функции, но нет кода теста к ней (или упоминания, что тест уже был написан ранее) — это нарушение инструкции.

* **Integration-тесты:** Используй `tests/` для реальных "дымных" тестов (Smoke Tests) на маленьких файлах (10-20KB), чтобы убедиться, что пайплайн работает end-to-end.

Тесты могут быть написаны как на скриптовых языках (bash, bats, fish), так и на rust.

В файле [Justfile](Justfile) присуствует короткие аллиасы для запуска различных видов тестов через `just`.

### Стандарты CLI и Справки

Для всех бинарников проекта (`0k`, `0k-core`) действуют следующие правила оформления справки (`--help`):

1. **Доступность справки:**
    * `binary <command> --help` (или `-h`) — выводит справку по конкретной команде.
